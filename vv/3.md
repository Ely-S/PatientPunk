# V3. Reddit Thread Reconstruction

## Verdict
Partial. Static review shows parent IDs are preserved through transform/import and used for upstream drug inheritance and immediate-parent classifier context, but the required orphan counts, chain sampling, and one/two/unlimited depth comparison are not implemented as auditable outputs.

## Static Evidence
- `docs/RCT_historical_validation/VV_PLAN.md:82-100` defines V3 checks for orphan parent counts, 50 sampled reply chains, ancestor-only drug inheritance, question-only parent context, and one/two/unlimited upstream-depth comparison.
- `Scrapers/transform_arctic_shift.py:77-88` emits Arctic Shift comment IDs as `t1_<id>` and preserves raw `parent_id`; `Scrapers/transform_arctic_shift.py:142-163` groups comments under submissions by `link_id` and skips comments whose submission is absent.
- `Scrapers/transform_arctic_shift.py:186-189` sorts comments by `created_utc`; this supports chronological storage but does not explicitly validate parent-before-child order.
- `src/import_posts.py:70-84` inserts submissions with `parent_id=NULL` and comments with `comment.get("parent_id")`; `src/import_posts.py:96-100` nulls dangling parent IDs after insert.
- `schema.sql:14-18` defines `posts.parent_id TEXT REFERENCES posts(post_id)`, but `schema.sql:26-27` only indexes user/date, not `parent_id`.
- `src/pipeline/extract.py:65-78` computes upstream drug context recursively from `id_to_parent`, with `max_depth=None` meaning unlimited and duplicate ancestor drugs de-duplicated in ancestor order.
- `src/pipeline/extract.py:81-125` loads `post_id`, `parent_id`, text, and recursively resolves a comment's root submission title through parent links.
- `src/pipeline/extract.py:150-157` and `src/pipeline/extract.py:173-182` include entries with direct or inherited drug mentions while excluding `is_only_questions(...)`, so question-only parents can contribute context but should not themselves enter tagged downstream classification.
- `src/utilities/__init__.py:28-46` exposes `max_upstream_depth` and validates it; `src/run_sentiment_pipeline.py:83-109` wires `--max-upstream-depth` and `--max-upstream-chars` into pipeline config.
- `src/pipeline/classify.py:172-190` builds a parent text lookup from tagged entries and backfills missing immediate parents from SQLite, including parents filtered out during extraction.
- `src/pipeline/classify.py:54-61`, `src/pipeline/classify.py:84-90`, and `src/pipeline/classify.py:247-284` include only the immediate parent as `Replying to` context in prefilter and classification prompts.
- `src/prompts/intervention_config.py:58-69` instructs the prefilter to use `Replying to` for short personal replies; `src/prompts/intervention_config.py:139-153` instructs classification to use upstream text only as context and require the reply itself to carry signal.
- `docs/RCT_historical_validation/_build_paper_figures.py` has no static references to `parent_id`, `drugs_context`, `Replying to`, or upstream-depth audit output.

## Gaps and Risks
- Dangling parent IDs are nulled in `src/import_posts.py` without recording pre-clean counts, affected IDs, or per-database orphan rates, so the orphan-cleaning step is not auditable after import.
- Recursive title and upstream-drug traversal have no explicit cycle guard beyond assuming valid Reddit parent trees; malformed imported data with a parent cycle could recurse indefinitely.
- Classifier prompt context is limited to the immediate parent text, while drug context extraction can inherit from unlimited ancestors. Deep context can therefore decide which drug is classified without showing the full chain to the classifier.
- `parent_id` has a foreign key but no dedicated index, which may make parent-chain audits or joins slower on large databases.
- Static code review found no implementation of random 50-chain sampling, no cross-thread validation report, and no automated comparison of one-, two-, and unlimited-depth classification outcomes.
- Transform grouping by `link_id` ensures comments are attached to the same submission, but there is no static validation that every preserved `parent_id` belongs to that same thread before import.

## Suggested Follow-up Checks
- Before import, count comments whose `parent_id` is missing from the union of submission/comment IDs in the Arctic Shift transform output, and write both counts and example IDs to an audit artifact.
- After import, run: `SELECT COUNT(*) FROM posts p WHERE p.parent_id IS NOT NULL AND NOT EXISTS (SELECT 1 FROM posts q WHERE q.post_id = p.parent_id);` and confirm zero dangling parents.
- Quantify nulled parents by importing to a temporary table or adding an audit query before cleanup: `SELECT COUNT(*) FROM posts WHERE parent_id IS NOT NULL AND parent_id NOT IN (SELECT post_id FROM posts);`.
- Sample reply chains with a recursive CTE from 50 comments stratified by depth; verify each ancestor exists, stays under the same root `t3_...` submission, and has monotonically earlier or equal timestamps than its child.
- Re-run extraction/classification on a fixed sample with `--max-upstream-depth 1`, `--max-upstream-depth 2`, and default unlimited depth; compare drug counts, prefilter pass rates, and final responder/non-responder labels.
- Sample question-only parent posts with child replies inheriting `drugs_context`; confirm the parent is absent from final `treatment_reports` while the child receives the intended drug context.
