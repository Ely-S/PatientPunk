# V7. Deduplication Audit

## Verdict
Partial. The final figure/table path enforces one retained row per user-drug pair with the documented sentiment and signal-strength priority, including cross-database naltrexone user deduplication, but the repository does not statically implement the required independent raw/unique count audit or quantify class changes caused by deduplication.

## Static Evidence
- `docs/RCT_historical_validation/VV_PLAN.md:173-200` defines V7 expected unique-user counts, retained-report priority, and acceptance criteria requiring one final row per user-drug pair plus reported deduplication effects.
- `docs/RCT_historical_validation/_build_paper_figures.py:63-82` defines `SENT_RANK = {"positive": 3, "mixed": 2, "neutral": 1, "negative": 0}` and `SIG_RANK = {"strong": 2, "moderate": 1, "weak": 0, "n/a": 0, None: 0, "": 0}`, then keys `best` by `(user_id, drug)` to keep one best report per user-drug.
- `docs/RCT_historical_validation/_build_paper_figures.py:142-146` calls `best_per_user_drug(...)` before filtering each non-naltrexone target drug and passes only the deduplicated sentiments into `_sentiment_breakdown`, so final `resp_df` rows are user-level for those drugs.
- `docs/RCT_historical_validation/_build_paper_figures.py:148-161` has special naltrexone handling that concatenates `year_2021` and `jan_2022`, filters naltrexone/LDN names, and keys `nalt_best` by `user_id`, deduping the user across both sources.
- `docs/RCT_historical_validation/_gen_analysis.py:42-61` and `docs/RCT_historical_validation/analysis.ipynb:39-53` duplicate the same `SENT_RANK`, `SIG_RANK`, and `(user_id, drug)` best-report selection logic.
- `docs/RCT_historical_validation/_gen_analysis.py:119-127` and `docs/RCT_historical_validation/analysis.ipynb:135-142` duplicate the naltrexone cross-database user-level dedup logic.
- `docs/RCT_historical_validation/_build_paper_figures.py:247-263` and `docs/RCT_historical_validation/_gen_analysis.py:191-200` hard-code Table 2 `unique_users` and `treatment_reports` values rather than querying them from the databases at report-generation time.
- `docs/RCT_historical_validation/README.md:91-99` describes one vote per user per drug and the intended best-report ranking in prose; `README.md:119-130` lists expected output counts for spot checking.
- `schema.sql:80-94` defines `treatment_reports` with `report_id` as the only primary key and indexes on post, drug, user, and run, but no uniqueness constraint preventing multiple reports for the same user-drug.
- `src/utilities/db.py:85-92` tracks existing `(post_id, drug_id)` pairs for incremental classification, not `(user_id, drug_id)` pairs; `src/utilities/db.py:103-111` inserts every accepted report, so duplicate user-drug reports can exist before analysis-level deduplication.

## Gaps and Risks
- Expected unique-user and raw treatment-report counts are hard-coded in generated tables and README prose; the static code does not independently query and compare raw report counts versus deduplicated user counts.
- No code was found that samples repeated user-drug pairs and verifies the retained report against the documented priority rule.
- No code was found that quantifies how many users changed responder class because of deduplication or whether deduplication materially changes responder percentages.
- The signal-strength tiebreak handles `n/a`, empty string, and Python `None`, but not case/whitespace variants such as `N/A`, `na`, or `" weak "` unless upstream normalization guarantees those values.
- `treatment_reports.user_id` is nullable in the schema; if null user IDs enter analysis, `best_per_user_drug` would collapse all null users for the same drug into one retained record.
- For non-naltrexone drugs, deduplication occurs before target-drug filtering but is keyed by canonical drug, so it is still one row per user-drug; however, aliases only dedupe together if canonicalization has already mapped them to the same `treatment.canonical_name`.

## Suggested Follow-up Checks
- Query raw and unique counts per target drug directly from each SQLite database, including `COUNT(*)`, `COUNT(DISTINCT tr.user_id)`, and duplicated `(user_id, canonical_name)` groups; compare against V7 expected counts.
- For naltrexone, union the two databases into one temporary table and compute `COUNT(*)`, `COUNT(DISTINCT user_id)`, duplicate users across both sources, and retained sentiments under the same rank tuple.
- Sample at least 30 duplicated user-drug pairs and print all candidate reports ordered by `(SENT_RANK, SIG_RANK)` beside the retained report for manual verification.
- Compute pre-dedup versus post-dedup responder percentages by drug, plus counts of users whose raw reports include both positive and non-positive labels.
- Quantify class changes by assigning each duplicated user-drug a raw majority/latest/first sentiment sensitivity label and comparing it with the retained best-report positive-vs-non-positive class.
- Add assertions in the build or a separate audit script that final deduplicated `n` values equal the expected unique-user counts: famotidine 207, loratadine 107, prednisone 176, paxlovid 153, colchicine 40, naltrexone 76.
